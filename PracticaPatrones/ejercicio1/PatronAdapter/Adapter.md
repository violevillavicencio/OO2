Se dispone de una clase ya implementada, Biblioteca, que cumple correctamente su función de exportar socios en formato JSON utilizando una clase específica (VoorheesExporter), pero surge la necesidad de cambiar la tecnología utilizada para generar ese JSON, incorporando librerías externas como JSON.simple o Jackson. La restricción fundamental es que no se puede modificar el código de la clase Biblioteca, lo cual obliga a buscar una solución que permita extender el comportamiento sin alterar el código existente. Este tipo de problema está directamente relacionado con el principio de abierto/cerrado (Open/Closed Principle), que establece que las clases deben estar abiertas a extensión pero cerradas a modificación.
El problema de fondo es que las nuevas librerías no comparten la misma interfaz que VoorheesExporter. Mientras que Biblioteca espera trabajar con un método exportar(List<Socio>) que retorne un String, las librerías externas tienen sus propias formas de construir estructuras JSON. Por lo tanto, existe una incompatibilidad entre la interfaz esperada por el cliente (Biblioteca) y la interfaz ofrecida por las librerías externas. Para resolver esta incompatibilidad se aplica el patrón de diseño Adapter.

El patrón Adapter tiene como objetivo permitir que dos interfaces incompatibles puedan trabajar juntas. Para ello, introduce una clase intermedia, llamada adapter, que actúa como traductor entre la interfaz que el cliente espera y la que realmente ofrece el componente externo. En este caso, Biblioteca actúa como cliente, ya que es quien utiliza el servicio de exportación sin conocer los detalles de implementación. La clase VoorheesExporter cumple el rol de target, es decir, define la interfaz que el cliente espera utilizar. Por otro lado, las librerías JSON.simple y Jackson son los adaptees, ya que ofrecen la funcionalidad deseada pero con una interfaz distinta e incompatible.
Los adapters se implementan mediante clases como JSONSimpleAdapter y JacksonAdapter, que extienden VoorheesExporter. De esta manera, respetan la interfaz esperada por Biblioteca, pero internamente delegan la responsabilidad de generar el JSON en las librerías correspondientes. Por ejemplo, JSONSimpleAdapter construye un JSONArray y múltiples JSONObject para representar los socios, mientras que JacksonAdapter utiliza un ObjectMapper para serializar directamente la lista de objetos. En ambos casos, el método exportar(List<Socio>) es sobrescrito para adaptar la lógica de la librería externa al formato esperado.

El funcionamiento del sistema es completamente transparente para la clase Biblioteca. Desde su punto de vista, sigue trabajando con un objeto de tipo VoorheesExporter. Sin embargo, gracias al uso de polimorfismo, ese objeto puede ser en realidad una instancia de JSONSimpleAdapter o JacksonAdapter. Esto permite cambiar dinámicamente la estrategia de exportación sin modificar el código del cliente, simplemente inyectando una nueva implementación mediante el método setExporter. En tiempo de ejecución, cuando Biblioteca invoca exportarSocios(), la llamada se delega al adapter correspondiente, que se encarga de traducir la operación hacia la librería externa y devolver el resultado en el formato esperado.
